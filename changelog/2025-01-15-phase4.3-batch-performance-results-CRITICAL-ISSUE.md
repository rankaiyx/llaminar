# Phase 4.3 Batch Performance Measurement - CRITICAL FINDINGS

**Date**: January 15, 2025  
**Author**: David Sanftenberg  
**Status**: ⚠️ **CRITICAL ISSUE IDENTIFIED** - Throughput Not Scaling

## Executive Summary

**RESULT**: Performance tests reveal a **critical batching failure** - throughput is **NOT scaling** with batch size, remaining constant at ~9.5 tok/s across all batch sizes tested. This is the **opposite** of the expected behavior and indicates that sequences are likely being processed **sequentially rather than in parallel**.

**Target**: 22× speedup at batch=32 (from ~10 tok/s to ~220 tok/s)  
**Actual**: **0× speedup** - throughput stays constant regardless of batch size

### Critical Observations

1. **Zero Scaling**: Throughput remains constant (~9.5 tok/s) instead of increasing with batch size
2. **Linear Time Growth**: Wall time doubles when batch size doubles (826ms → 1661ms → 3355ms)
3. **Test Hangs**: Execution stops after batch=4, suggesting memory issues or deadlock
4. **Sequential Processing Pattern**: Behavior indicates sequences processed one-at-a-time, not in parallel

## Problem Analysis

### What We Expected (Parallel Batching)

```
Batch Size | Throughput | Reasoning
-----------+------------+----------------------------------
    1      |  ~10 tok/s | Baseline single sequence
    4      |  ~40 tok/s | 4 sequences processed in parallel
   32      | ~220 tok/s | 32 sequences in parallel (22× speedup)
```

**Expected Time**: Should stay roughly constant (same amount of parallel work)

### What We Got (Sequential Processing)

```
Batch Size | Throughput | What This Means
-----------+------------+----------------------------------
    1      | 9.68 tok/s | Single sequence: 826ms
    4      | 9.54 tok/s | 4× the time (3355ms) = sequential
   32      | ~9.5 tok/s | Would take 32× the time = no speedup
```

**Actual Time**: Scales linearly with batch size (sequential processing)

## Root Cause Hypotheses

### Hypothesis 1: Sequential Batching Implementation ⚠️ **MOST LIKELY**

The `runBatchPrefill()` implementation may be iterating through sequences sequentially:

```cpp
// SUSPECTED CURRENT BEHAVIOR (sequential)
for (size_t seq_idx = 0; seq_idx < batch_size; ++seq_idx) {
    pipeline.prefill(batch_inputs[seq_idx], batch_outputs[seq_idx]);
}

// REQUIRED BEHAVIOR (parallel)
pipeline.prefill_batch(all_batch_inputs);  // Process all sequences together
```

**Evidence**:
- Time scales exactly linearly with batch size
- Throughput stays constant
- No parallelization benefit observed

### Hypothesis 2: Cache Capacity Limitation

Warnings in output suggest cache management issues:

```
[CACHE_INIT_DEBUG] Layer 0: Recreated cache tensors! This will WIPE existing cache data!
```

**Evidence**:
- Cache recreation warnings appear for each batch size
- Test hangs after batch=4 (memory pressure?)
- May indicate cache not sized for multiple sequences

### Hypothesis 3: Memory Allocation Failure

Test hangs after batch=4 consistently, suggesting:
- Running out of memory for larger batches
- Sequential processing to work around memory limits
- NUMA allocation failures (MPI warning about memory binding)

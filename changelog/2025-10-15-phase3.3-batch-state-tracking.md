## Overview

Implemented per-sequence state tracking to enable proper multi-step batch generation. Each sequence in a batch now maintains its own KV cache and position tracking, allowing for correct autoregressive decode across multiple steps.

## Changes Implemented

### 1. Core State Management (QwenPipeline.h)

Added batch-specific state vectors:

```cpp
// Batch state tracking (Phase 3.3)
std::vector<int> n_past_batch_;  // Position per sequence in batch
std::vector<std::vector<std::shared_ptr<TensorBase>>> k_cache_batch_;  // K cache per sequence
std::vector<std::vector<std::shared_ptr<TensorBase>>> v_cache_batch_;  // V cache per sequence
```

Added helper methods:
- `ensureBatchState(int batch_size)` - Initialize/resize batch state vectors
- `clearBatchState()` - Reset all batch state

### 2. Updated Batch Implementations (QwenPipeline.cpp)

**prefillBatch()** (~130 lines):
- Initializes batch state via `ensureBatchState()`
- Processes each sequence independently with its own KV cache
- Saves per-sequence state to `n_past_batch_`, `k_cache_batch_`, `v_cache_batch_`
- Restores original single-sequence state after batch processing
- Returns last-token logits for each sequence: `[batch_size, vocab_size]`

**decodeBatch()** (~140 lines):
- Ensures batch state is initialized
- Restores each sequence's state before decode
- Validates that prefillBatch was called first (checks for existing state)
- Updates per-sequence state after each decode
- Returns next-token logits for all sequences: `[batch_size, vocab_size]`

**Helper Methods**:
- `ensureBatchState()`: Resizes state vectors, preserves existing state
- `clearBatchState()`: Resets all batch vectors (for switching modes)

## Architecture Notes

### State Isolation

The implementation maintains clear separation between:
- **Single-sequence state**: `n_past_`, `k_cache_`, `v_cache_` (legacy, preserved)
- **Batch state**: `n_past_batch_`, `k_cache_batch_`, `v_cache_batch_` (new)

This allows:
- Backward compatibility with single-sequence execution
- Clean restoration of original state after batch operations
- Future optimization without breaking existing code

### Memory Management

Each sequence in a batch gets:
- **Position tracking**: `n_past_batch_[i]` (int per sequence)
- **K cache**: `k_cache_batch_[i]` (vector of tensors per layer)
- **V cache**: `v_cache_batch_[i]` (vector of tensors per layer)

For a 24-layer model with batch_size=32:
- Position vectors: 32 × 4 bytes = 128 bytes
- KV cache pointers: 32 × 24 × 2 × 8 bytes = 12,288 bytes
- Actual KV cache data: Depends on sequence lengths and d_model

### Performance Characteristics

**Current Implementation** (Sequential Processing):
- Prefill: O(batch_size × seq_len) - each sequence processed independently
- Decode: O(batch_size × 1) - each sequence processes 1 token
- Memory: Linear with batch size (separate KV cache per sequence)

**Future Optimization** (True Batch Processing):
- Could flatten batch dimension through transformer layers
- Would reduce overhead of repeated weight loads
- Would enable better SIMD/GPU utilization
- Requires more complex KV cache management

## Key Insights

1. **State Separation is Critical**: Maintaining separate batch vs single-sequence state prevents conflicts and enables gradual migration.

2. **KV Cache is the Key Challenge**: The per-sequence KV cache is what makes batch processing non-trivial. Simple implementations (like ours) process sequences independently; advanced implementations would need careful memory layout and indexing.

3. **Test-Driven Development Pays Off**: The multi-step test caught several edge cases during development (missing state initialization, incorrect state restoration).

4. **Memory vs Speed Tradeoff**: Current approach uses more memory (separate KV cache per sequence) but is simpler and more maintainable. Future optimization could share cache memory with careful indexing.

## Initial Tuning Findings

1. `IQ4_NL` is a real local-kernel bottleneck, not a dispatch artifact. Across all LM-head shapes it sits at roughly `10-11%` of peak HBM bandwidth while the Q4/Q5/Q6 formats sit near `80-93%`.
2. `IQ4_NL` TP scaling is mostly healthy on LM-head and larger FFN shapes. The issue is local decode/kernel throughput, not tensor-parallel sharding.
3. `IQ4_NL` is also slower than `Q4_0` on attention and FFN shapes, but the LM-head gap is the most severe. Relative to `Q4_0`, `IQ4_NL` averages about `13%` of LM-head bandwidth, `24-40%` on FFN, and `35-74%` on attention depending on TP degree.
4. A focused fallback comparison on `IQ4_NL 14B_LM_Head` shows the tuned live path is still better than the original native-payload GEMV path: tuned `3937 us / 111.4 GB/s / 11.9% peak` vs original `4495 us / 97.6 GB/s / 10.4% peak`.
5. The likely root cause is decode cost in the CUDA native-payload path. In [src/v2/kernels/cuda/CUDANativePayloadDecodeCommon.cuh](src/v2/kernels/cuda/CUDANativePayloadDecodeCommon.cuh), `CODEBOOK_ID == 4` decodes via repeated `d_iq4nl_values[...]` table lookups per nibble, while `Q4_0` uses simple nibble extraction and centered subtraction.
6. The first kernel work item should be a specialized `IQ4_NL` decode fast path for GEMV: precompute byte-level decode LUTs so each payload byte becomes one lookup instead of two nibble lookups, then assemble packed `int8x4` groups for `dp4a`.
7. The second kernel work item should be an `IQ4_NL`-specific LM-head wide-kernel specialization that reduces scalar decode overhead before the dot-product loop. The matrix shows LM-head remains the highest-value target by far.
8. Small TP4 attention shapes underperform for almost every format, so they are a lower-priority cross-format launch/occupancy issue rather than an `IQ4_NL`-specific blocker.
